Какой минимальный набор метрик вы выведите в мониторинг и почему?

Ответ: для подобного проекта, будут важны:
- показатели скорости чтения и записи данных а так же заполнения дискового пространства (метрики:disk_free_GB, disk_read_write_MBps), 
- количество ошибок протокола http, время отклика пользователей и количество запросов (метрики:http_status_code, http_error_rate_%, http_response_time_ms, http_requests_per_sec), 
- загруженность и температура ЦПУ (метрики:cpu_usage_%, cpu_temp_C).
- критические ошибки в логах (метрика:error_log_count).

2. Менеджер продукта посмотрев на ваши метрики сказал, что ему непонятно что такое RAM/inodes/CPUla. Также он сказал, что хочет понимать, 
насколько мы выполняем свои обязанности перед клиентами и какое качество обслуживания. Что вы можете ему предложить?

Ответ: в данном случае, стоит перевести технические термины в удобные для менеджера понятия, связать метрики с конкретными рисками для проекта. 
Качество предоставляемых услуг следует изложить в показателях: 
SLI (Service Level Indicator):
Время ответа API (http_response_time_ms).
Доля успешных запросов (success_rate_%).
Доступность (uptime_%).

SLO (Service Level Objective):
«99,9 % запросов должны отвечать за <1 с».
«Не более 0,1 % ошибок в день».
«Простой не дольше 5 минут в неделю». 

Показатели следует предоставить в виде линейного графика с конкретными значениями.

3. Вашей DevOps команде в этом году не выделили финансирование на построение системы сбора логов. Разработчики в свою очередь хотят видеть все ошибки, 
которые выдают их приложения. Какое решение вы можете предпринять в этой ситуации, чтобы разработчики получали ошибки приложения?

Ответ: следует выстроить работу на бесплатных сервисах:

- syslog‑ng / rsyslog Назначение: централизованный сбор логов с серверов на один узел.

- rsync / scp Назначение: копирование лог‑файлов с серверов на центральный узел.

- grep / awk / sed Назначение: поиск и фильтрация строк в логах.

- jq Назначение: анализ JSON‑логов.

- nc (netcat) / socat Назначение: простейшая передача логов по сети в реальном времени.

- cron + mail Назначение: автоматические оповещения по email при находке ошибок.

- logrotate Назначение: ротация и очистка старых логов.

- GoAccess Назначение: веб‑интерфейс для анализа логов (например, HTTP‑запросов).

- SSH + CLI Назначение: прямой доступ разработчиков к логам на сервере. Подключение по SSH и работа с grep/jq/less.

5. Опишите основные плюсы и минусы pull и push систем мониторинга.

Ответ:
## pull-система

# Плюсы:
- Простота настройки и управления. Одна точка конфигурации — сервер мониторинга; не требуется установка агентов на целевых системах.
- Интеграция с service discovery. Легко работает с механизмами автоматического обнаружения сервисов (Kubernetes, Consul и др.): 
новые цели добавляются в мониторинг без ручного вмешательства.
- Надёжность и контроль. Сервер видит, когда цель недоступна (метрика up), что упрощает диагностику сбоев.
- Гибкость конфигурации. Можно тонко настраивать интервалы сбора и список собираемых метрик для каждой цели.
- Актуальность данных. Регулярный опрос гарантирует, что метрики обновляются с заданной частотой.

# Минусы:
- Масштабируемость. При большом числе целей растёт нагрузка на сервер мониторинга и сеть; требуется больше ресурсов.
- Сложность конфигурации в крупных системах. Управление файлами конфигурации (например, prometheus.yml) может стать громоздким.
- Нагрузка на цели. Частый опрос увеличивает нагрузку на мониторинговые сервисы и сервер мониторинга.
- Риск потери данных при сбоях. Если сервер мониторинга недоступен или есть проблемы с сетью, данные за период простоя не соберутся
(но могут быть восстановлены при восстановлении связи).
- Требование открытости endpoints. Цели должны быть доступны для опроса, что может быть проблематично в защищённых сетях.

## push-система

# Плюсы:
- Сниженная нагрузка на цели. Сервисы отправляют метрики только когда готовы, без регулярного опроса.
- Гибкость для краткосрочных задач. Идеальна для batch‑процессов, кратковременных заданий, которые не успевают быть опрошенными pull‑системой.
- Эффективность ресурсов. Меньше сетевого трафика и CPU‑нагрузки, так как данные отправляются только при наличии.
- Работа в ограниченных сетях. Подходит для сред, где цели находятся за фаерволами и не могут быть опрошены извне.
- Буферизация данных. Промежуточные шлюзы (вроде Pushgateway) могут временно хранить метрики, если сервер мониторинга недоступен.

# Минусы:
- Риск потери данных. Если цель недоступна в момент отправки или сервер не принимает данные, метрики могут быть утеряны.
- Зависимость от агентов. Требуется установка и поддержка агентов на целевых системах.
- Единая точка отказа. Промежуточные шлюзы (например, Pushgateway) могут стать узким местом или точкой отказа.
- Сложность мониторинга здоровья целей. Трудно определить, когда цель упала, так как нет регулярного опроса (нет метрики up).
- Накопление устаревших данных. Шлюзы могут хранить устаревшие метрики, если их не очищать вручную.
- Менее предсказуемая актуальность данных. Данные поступают нерегулярно, что усложняет анализ трендов.

9. Изучите список telegraf inputs. Добавьте в конфигурацию telegraf следующий плагин - docker:

```
[[inputs.docker]]
  endpoint = "unix:///var/run/docker.sock"
```

Дополнительно вам может потребоваться донастройка контейнера telegraf в docker-compose.yml дополнительного volume и режима privileged:

```
  telegraf:
    image: telegraf:1.4.0
    privileged: true
    volumes:
      - ./etc/telegraf.conf:/etc/telegraf/telegraf.conf:Z
      - /var/run/docker.sock:/var/run/docker.sock:Z
    links:
      - influxdb
    ports:
      - "8092:8092/udp"
      - "8094:8094"
      - "8125:8125/udp"
```
